### 1. `[LEARN:env]` rdrobust R 3.0 vs STATA rdrobust v18 ‚Äî `e(h_l)` mismatch

**Rule:** R `rdrobust::rdbwselect` `bws[1,1]` (h column) ‚âÝ STATA `e(h_l)` for most outcomes when bwselect=mserd. Only farm_income matches; for other outcomes the R `b` column also doesn't match. Both packages by the same authors but version-divergent.

**Why:** Cattaneo-Calonico-Titiunik suite has had multiple selector-default changes across versions; STATA v18 and R 3.0 are not coordinated. Same MSE-optimal target, different sieve-of-Eratosthenes implementation details.

**How to apply:** Any STATA ‚Üí R rdrobust replication. For tolerance-bound replication, do NOT trust R's MSE-optimal as a substitute for STATA's; either hardcode STATA's reported h*[outcome] verbatim OR document the divergence explicitly. P3 work: investigate which selector internal differs.

### 2. `[LEARN:methods]` STATA log line attribution ‚Äî Step 2 vs Step 5 rwolf2

**Rule:** STATA `02_analysis.do` has TWO reghdfe blocks: Step 2 (per-outcome MSE bandwidth, lines 159-176) AND Step 5 rwolf2 (common hA_min for family-wise correction, lines 263-275). Both emit reghdfe outputs to log. Distinguishing them requires checking per-outcome N ‚Äî Step 2 yields different N per outcome (different bandwidth); Step 5 yields uniform N (common bandwidth).

**Why:** Initial replication targeted Step 5 (common hA_min=3268 for all 4 outcomes), got 1/4 PASS for Spec A (only the binding outcome farm_income matched). Re-reading the log showed Step 2's per-outcome N pattern (6072/6894/7334/6135) ‚Äî that's the actual ground truth for individual-outcome reghdfe.

**How to apply:** Any STATA ‚Üí R replication of multi-outcome DiD-RD with multiple bandwidth blocks. ALWAYS check per-outcome N in the log to identify which block produced each reported coefficient. Don't trust line proximity (e.g., "lines 394/442/490/538 are 50 lines apart" doesn't imply same block).

### 3. `[LEARN:methods]` STATA `%8.0f` display rounding ‚Üí integer h vs float h actual

**Rule:** STATA `di as txt ... %8.0f e(h_l)` rounds the float bandwidth to an integer for display, but the underlying `if abs(rv_2018) <= e(h_l)` uses the float. The float can be `3303.49` (displayed as "3303") or `3303.51` (displayed as "3304"). R's `|rv| <= 3304` sample then differs from STATA's `|rv| <= 3303.49`.

**Why:** Boundary observations get included differently. For a sample of ~6000 obs, 1-5 boundary obs change the regression sample ‚Üí coefficient changes by a few thousand KRW (0.04-1% relative diff).

**How to apply:** When hardcoding STATA-reported bandwidths in an R replication, treat the reported integer as ¬± 2 search range. Search h ‚àà {h_int‚àí2, ..., h_int+2}, pick the integer matching STATA's reported regression N (also in the log). Achieves sub-1-KRW agreement. P2 used this technique to go from 6/8 PASS ‚Üí 8/8 PASS.

### 4. `[LEARN:env]` `sandwich::vcovBS` √ó `fixest` incompatibility

**Rule:** `sandwich::vcovBS` for `fixest::feols` fits dispatches to `vcovBS.default`, which (a) does NOT support `type='wild'` (only `xy`, `jackknife`, `fractional`); (b) on `fixest` refit hits a `data` argument conflict: sandwich's `update()` mechanism passes data through a function symbol that fixest interprets as a function. Result: any vcovBS call fails for fixest, whether `type='wild'` or `type='fractional'`.

**Why:** sandwich's `vcovBS.lm` method supports wild; the `vcovBS.fixest` method does not exist; `vcovBS.default` is type-limited AND refit-broken for fixest's data-handling style.

**How to apply:** For Wild cluster bootstrap with fixest fits, you have three options:
- `fwildclusterboot::boottest()` ‚Äî proper Wild cluster bootstrap, but requires gfortran on R 4.5+ (see `[LEARN:env]` 2026-05-06 ‚Äî pragmatic fallback was sandwich, which now also fails).
- Manual: refit with `lm(y ~ ... + factor(hh_id) + factor(year))` (slow, memory-heavy on N=14k √ó 4000 households), then `sandwich::vcovBS(fit, type='wild')`.
- `boot` package with manual Wild-multiplier draws and per-iteration feols refit.

P2 fell back to `fixest`'s built-in cluster-robust SE (Liang-Zeger CR1), which exactly matches STATA `reghdfe vce(cluster hhid_num)`. Wild bootstrap deferred to P3 (`04_robust.R`).

### Bonus (lesser candidate): `[LEARN:tooling]` `quality_score.py` false-positive regex

**Rule:** `scripts/quality_score.py` `check_hardcoded_paths` uses regex `["'][/\\]|["'][A-Za-z]:[/\\]` to detect absolute paths. This produces false positives on:
- Any quoted string starting with `\` (e.g., LaTeX templates `"\\begin{table}..."`)
- Any quoted escape sequence `"\n"`, `"\t"`, etc.

**Why:** The regex treats `"\\` as a quote followed by a path-separator backslash. In R, `"\\` is the start of an escaped backslash literal ‚Äî common in LaTeX strings and Windows path placeholders.

**How to apply:** Either refine the regex to require a path-looking pattern after the slash (e.g., `["'][/\\][A-Za-z]+/`), or add an allowlist of escape sequences (`\\n`, `\\t`, `\\\\`, `\\begin`, `\\end`, etc.). P2 hit 4 false positives in `02_descriptive.R` (lines 188/190) and `03_did_rd.R` (lines 459/469/506/508). Lower priority than the 4 main candidates; P3+ infrastructure work.
